## 一、DBS 星展银行 · 实时交易监控与告警平台（2024年10月 ~ 2025年02月）

### 项目概述
DBS 零售银行需要一个实时交易监控系统，对高频交易流水进行风险规则匹配与实时告警。日均处理交易量约 500 万笔，要求告警延迟不超过 3 秒。

### 我的职责
- 独立负责后端架构设计与核心模块开发
- 使用 Kafka 消费交易流水，实现规则引擎实时匹配
- 使用 Redis 缓存风险规则与黑白名单，实现热更新
- 使用分布式锁（Redis + Redisson）实现告警去重，避免同一异常重复通知
- 对接邮件、短信、内部通讯工具多通道告警推送
- 编写单元测试与集成测试，保障规则引擎正确性

### 技术栈
Kafka、Spring Boot、Redis、Redisson、Spring Cloud Gateway、MySQL

### 常见问题与应答

**Q1: Kafka 如何处理高吞吐的交易流水？**
> 我们配置了 12 分区并行消费，每个消费者独立拉取分区数据。Producer 侧按交易 ID 哈希分区，保证同一笔交易相关事件进入同一分区，有序处理。Consumer 侧使用批量拉取（`max.poll.records=500`），减少网络开销。同时开启了手动提交 Offset，确保处理完成后才确认消费。

**Q2: 规则引擎如何实现？**
> 规则配置存储在 MySQL 中，启动时加载到 Redis 缓存。规则以 JSON 表达式存储，例如 `{"field": "amount", "op": "gt", "value": 50000}`。引擎在内存中解析 JSON 规则树，逐条匹配交易字段。规则变更时通过 Redis Pub/Sub 通知所有实例刷新本地缓存，实现热更新。

**Q3: 分布式锁如何保证告警不重复？**
> 使用 Redis + Redisson 的 `tryLock`，锁 Key 为 `alert:{ruleId}:{userId}:{date}`，超时时间 30 秒。获取锁成功后查询该时段是否已有告警记录，无则发送并记录，有则跳过。即使多实例同时处理同一异常交易，只有一个实例能获取锁并发送告警。

## 二、DBS 星展银行 · Wealth API 核心服务开发（2023年4月 ~ 至今）

### 项目概述
DBS 零售银行财富管理系统的核心 API 服务，提供资产查询、账户操作、投资交易等功能，服务数百万零售客户。日均 API 调用量约 2000 万次。

### 我的职责
- 参与核心财富 API 功能开发，负责数据库表设计与创建、RESTful API 接口实现
- 完成第三方系统（交易系统、账户系统）接口对接与联调
- 通过异步调用和多线程优化，接口响应时间降低 30%+
- 使用 Jenkins Pipeline 实现 CI/CD 自动化部署与测试
- 与新加坡、印度团队协作进行故障排查，保障系统 SLA 99.9%

### 技术栈
Spring Boot、Spring Cloud（Feign/GateWay/Sentinel）、Jenkins、TWS Job、Avaloq

### 常见问题与应答

**Q1: 如何进行接口性能优化？**
> 主要从三方面入手：① 使用 `@Async` + 自定义线程池将非关键路径（如日志记录、通知推送）异步化，减少主线程阻塞；② 使用 Redis 缓存热点数据（如汇率、产品列表），设置合理过期时间，缓存命中率达 85%+；③ 数据库查询使用索引优化 + 慢 SQL 治理，使用 `EXPLAIN` 分析执行计划，避免全表扫描。

**Q2: Spring Cloud Gateway 在项目中怎么用的？**
> 作为 API 网关统一入口，实现路由转发、鉴权、限流。鉴权层解析 JWT Token 后将用户信息通过 Header 透传给下游服务；限流使用 Sentinel 集成，按 API + 用户维度设置 QPS 阈值，超过阈值返回 429。

**Q3: 跨区域团队协作怎么保障质量？**
> 我们采用 Git Flow 分支策略，Feature 分支开发 → PR 提交 → Code Review（至少 1 位 Reviewer）→ Jenkins 自动构建 + 单元测试 → 合并到 Develop。每周与新加坡团队有一次同步会议，关键需求使用 Confluence 文档记录，避免理解偏差。

## 三、DBS 星展银行 · 客户数据聚合服务（2025年01月 ~ 2025年04月）

### 项目概述
为 DBS 客户经理提供统一的客户 360° 视图，聚合来自多个内部系统（核心银行系统、CRM、交易系统）的客户数据，支持实时查询与批量导出。

### 我的职责
- 负责数据聚合层的架构设计与开发
- 使用 Feign 调用多个内部微服务获取客户数据（资产、持仓、交易记录、KYC 信息）
- 使用 CompletableFuture 并行调用多个服务，将客户视图查询响应时间从 800ms 降至 200ms
- 使用 Redis 缓存客户基础信息（TTL 5 分钟），避免频繁跨服务调用
- 实现大数据量 Excel 导出功能，使用 EasyExcel 流式写入，支持 10 万+行数据的导出
- 使用 Spring Security + JWT 实现细粒度权限控制（不同角色查看不同客户维度）

### 技术栈
Spring Boot、Spring Cloud Feign、Redis、EasyExcel、Spring Security、JWT、MySQL

### 常见问题与应答

**Q1: 并行调用多个服务是怎么实现的？**
> 使用 `CompletableFuture.supplyAsync()` 将每个服务调用提交到自定义线程池（核心线程数 = 服务数 * 2），然后使用 `CompletableFuture.allOf().join()` 等待所有结果返回。设置了超时时间 2 秒，超时服务返回降级数据。相比串行调用，5 个服务从 800ms 降到 200ms 左右。

**Q2: 大数据量 Excel 导出怎么处理？**
> 使用 EasyExcel 的流式写入模式，配合 MyBatis 游标查询（`ResultHandler`），每次从数据库读取 1000 行处理，写完后释放内存，避免一次性加载全量数据导致 OOM。经过测试，10 万行数据导出约 15 秒，内存占用控制在 200MB 以内。

**Q3: 细粒度权限怎么设计的？**
> 使用 RBAC 模型，用户 → 角色 → 权限。权限粒度到字段级别，例如客户经理角色可以查看客户资产和持仓，但看不到 KYC 敏感信息；合规角色可以查看全部。权限判断使用 Spring Security 的 `@PreAuthorize` + 自定义 `PermissionEvaluator` 实现。

## 四、Nomura 野村证券 · E-Platform 网上银行（2021年11月 ~ 2023年4月）

### 项目概述
日本野村证券面向海外高净值客户的网上银行系统，提供全球资产查询、账户管理和订单管理功能。支持多币种、多语言。

### 我的职责
- 负责后端功能模块的接口开发，包括资产查询、账户详情、订单管理
- 参与需求分析，独立完成数据库表设计、工作流设计
- 开发定时任务调度模块（Quartz），实现资产报表自动生成与推送
- 使用 Redis 缓存高频查询数据，资产列表查询性能提升 40%+
- 编写 Jenkins 部署脚本，实现多环境一键部署

### 技术栈
Spring MVC、JPA、Redis、Jenkins、Quartz、Avaloq

### 常见问题与应答

**Q1: JPA 和 MyBatis 怎么选择使用？**
> 这个项目使用 JPA 是因为 Nomura 有严格的 DDL 规范，表结构由 Avaloq 平台统一管理，实体关系清晰，JPA 的自动映射和级联操作可以减少很多模板代码。但复杂查询使用 `@Query` 原生 SQL，避免 JPA 生成的 SQL 性能不达标。在国内项目我更倾向 MyBatis/MyBatis-Plus，SQL 可控性更强。

**Q2: 定时任务怎么保证不重复执行？**
> 使用 Quartz 的集群模式，多个实例共享同一套 `QRTZ_*` 表。Quartz 通过数据库行锁（`SELECT ... FOR UPDATE`）保证同一个 Job 在同一时刻只有一个实例执行。我们为关键任务配置了 `@DisallowConcurrentExecution`，防止同一任务并发。

**Q3: 多币种场景下金额怎么处理？**
> 所有金额存储时统一使用 `DECIMAL(20,4)`，避免浮点数精度问题。计算时使用 `BigDecimal`。前端展示时根据币种动态选择小数位数（如日元展示整数，美元展示 2 位小数）。

## 五、Nomura 野村证券 · 批量数据ETL与报表系统（2022年03月 ~ 2023年01月）

### 项目概述
为 Nomura 运营团队提供批量数据处理与报表生成平台，每日从 Avaloq 核心系统导出交易数据，经过清洗、转换、聚合，生成多维度报表（损益报表、交易量统计、客户活跃度分析等），并通过邮件和内部门户分发。

### 我的职责
- 独立负责后端ETL流程设计与开发
- 使用 DynamicDatasource 实现从多个数据源（Avaloq/Oracle、PostgreSQL、CSV 文件）动态切换读取数据
- 使用 Spring Batch 编排批量处理流程（Reader → Processor → Writer），支持分片并行处理（Partitioner）
- 实现大数据量报表的 Excel 生成，使用 EasyExcel 模板填充 + 流式写入
- 使用 Seata 分布式事务保证跨数据源操作的一致性
- 开发 Jenkins 定时任务，每日凌晨 2:00 自动触发 ETL 流程
- 编写数据校验规则，异常数据自动写入错误表并邮件通知运维

### 技术栈
Spring Boot、Spring Batch、DynamicDatasource、Seata、EasyExcel、Jenkins、Oracle、PostgreSQL

### 常见问题与应答

**Q1: Spring Batch 怎么处理大数据量？**
> 使用分片（Partitioning）机制，按日期或客户 ID 哈希将数据分成多个分片，每个分片独立线程处理。例如处理 100 万条交易记录，分成 10 个分片，每个分片处理 10 万条，10 个线程并行执行，总耗时从串行的 30 分钟降到 5 分钟左右。Chunk 大小设为 1000，每次提交一次事务，既保证事务原子性又不会过份频繁提交。

**Q2: DynamicDatasource 怎么解决动态切换数据源？**
> 使用 `@DS` 注解在方法级别声明数据源。Avaloq 主数据源作为默认数据源，PostgreSQL 报表库在写入时切换。数据源配置存储在配置中心，运行时可以通过 `DynamicRoutingDataSource` 动态添加新的数据源，无需重启。底层使用 ThreadLocal 保证线程安全。

**Q3: ETL过程中出现错误怎么处理？**
> 采用 Skip 策略：在 Spring Batch 中配置 `skipLimit` 和 `skipPolicy`，单条数据异常（如数据格式错误）跳过并记录到错误表，不中断整批处理。对于致命错误（如数据库连接失败），配置了重试机制（最多 3 次），重试失败后记录日志并发送邮件告警。采用补偿机制，通过检查错误表重新处理失败数据。

## 六、DBS 星展银行 · 内部API网关升级迁移（2024年03月 ~ 2024年08月）

### 项目概述
DBS 内部老旧 API 网关（基于 Netflix Zuul）需要升级迁移到 Spring Cloud Gateway，涉及 50+ 微服务的路由迁移、鉴权适配、限流策略迁移。要求平滑迁移，业务零中断。

### 我的职责
- 参与技术方案设计，主导路由配置迁移脚本开发
- 使用 Spring Cloud Gateway 替代 Zuul，重新配置 50+ 路由规则
- 实现自定义 GlobalFilter 完成 JWT 鉴权、请求日志记录、链路追踪（TraceId）
- 集成 Sentinel 实现网关层限流与熔断降级
- 实现灰度发布策略：按 Header 中的 `version=v2` 将流量逐步切换到新网关
- 编写自动化回归测试脚本，验证新旧网关响应一致性
- 迁移后系统吞吐量提升 40%，平均延迟降低 25%

### 技术栈
Spring Cloud Gateway、Spring Security、Sentinel、JWT、Redis、Docker、Jenkins

### 常见问题与应答

**Q1: 网关迁移怎么做到业务零中断？**
> 采用灰度发布策略：新网关部署在独立端口（8081），上游 Nginx 通过 `map` 指令根据 Header `X-Gateway-Version` 分流。初期 10% 流量到新网关，监控无异常后逐步提升到 50%、100%。同时新旧网关并行运行 2 周，出问题可秒级回滚。路由配置使用 Git 管理，变更需要 Code Review。

**Q2: Spring Cloud Gateway 和 Zuul 有什么区别？**
> Spring Cloud Gateway 基于 WebFlux（响应式），使用 Netty 作为容器，非阻塞 I/O，性能更好。Zuul 1.x 基于 Servlet，是阻塞式 I/O，线程模型不如 Gateway 高效。在我们的压测中，Gateway 相同机器配置下 QPS 比 Zuul 高约 40%。另外 Gateway 的路由配置更灵活，支持 Java DSL 和 YAML 两种方式。

**Q3: 网关层的限流怎么实现的？**
> 使用 Sentinel 集成 Spring Cloud Gateway 的 `SentinelGatewayFilter`。按路由 + 用户维度限流，例如 `/api/asset` 每用户每秒 50 次。限流规则存储在 Sentinel Dashboard，支持动态修改。超过阈值时返回 HTTP 429 + 友好提示信息，不影响其他路由的正常服务。
